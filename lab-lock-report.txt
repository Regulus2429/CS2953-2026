Memory Allocator
原来的空闲物理页是通过单一链表和单一全局自旋锁实现的，所以锁竞争激烈。解决方案在Hint中给出，
也就是每个 CPU 维护自己专属的空闲链表和锁。当某个 CPU 可用内存耗尽，就从其他 CPU 链表中
窃取空闲页。实现的时候，可以参考原本的代码，加上获取 cpuid 的部分即可。需要注意的是，
调用 cpuid 之前需要关闭中断，否则线程可能会在获取 id 之后被调度到其他物理核心，导致跨核心操作。
由于涉及到的锁较多，写完后最好检查获取和释放的锁是否写错。
容易写错的是偷取其他 CPU 物理页的代码。最简单的实现方式就是一次只偷一页，我经过测试发现这样
可以通过测试样例。但是一次只获取一个页可能会导致反复窃取，所以一种优化方式是一次性窃取多个页。
实现时会复杂一些，因为此时还需要向当前 CPU 的 freelist 中插入窃取的页(只窃取一个页就不用)。
易错点在于当将窃取的链表插入时，需要让插入的链表的尾部连接当前的 freelist，因为 freelist 
可能在刚刚被别的核心插入了可用的页。经过测试，这样确实效果更好，也可以通过测试样例。
最后，由于有些锁的释放是在if条件后释放的，务必记住在if条件外也得释放！

Buffer Cache
这里的实现比上一问难，需要自行引入哈希表结构。思路就是根据块号把块分到哈希表的不同桶内，每个桶
都有一定的空闲块可供使用，如果当前桶内没有多余的空闲块用于缓存，也需要去别的桶内窃取。
我初次实现bget的时候，觉得不需要加上全局锁就可以实现窃取块。但是我忽略了一个问题：单一缓存副本。
如果不使用全局的锁，那么在释放当前桶去别的桶找空闲块的间隙中，其余并发线程可能已经将目标块移入
当前桶中，导致最后出现一个块的两个副本。所以去别的块中窃取之前必须加上全局锁。然后的一个易错点，
是加上全局锁之后还需要再扫一遍当前桶，因为获取全局锁的间隙中可能这个块被加入了桶中，需要再检查
一遍。然后从别的桶中偷取块会涉及到两个桶中链表的元素插入删除，以及两个锁。为了避免死锁，最好
不要同时获取不同桶的锁(比如在遍历别的桶前，先释放当前桶的锁)，即使获取也必须保持一致的获取顺序
和释放顺序。最后，题目没有要求实现 LRU，所以在brelse中不用再实现此逻辑。
别的函数实现都比较简单，没什么易错点，重点关注 bget 就好。
我在编译的时候发现如果写 const int NBUCKET 会触发 variably modified array at file scope 
编译错误。C语言中必须使用宏定义 #define NBUCKET 来声明编译期间常数数组的大小。
